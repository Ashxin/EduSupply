## 1. Git & GitHub Fundamentals

**Why an empty GitHub repo (no README/.gitignore/license at creation)?**
Git history is a chain of commits, each pointing to the one before it. If GitHub starts with its own initial commit (from a README) and your local repo starts a *separate* chain via `git init`, the two histories don't share a common ancestor — causing conflicts on first push. Starting empty on both sides avoids this.

**Core commands used to connect local → remote:**
```bash
git init                                   # turns a folder into a git repo
git add .                                  # stages files for commit
git commit -m "message"                    # commits staged files with a message
git branch -M main                         # renames default branch to 'main'
git remote add origin <repo-url>           # links local repo to GitHub remote
git push -u origin main                    # first push; -u sets upstream tracking
git push                                   # subsequent pushes (no -u needed once tracking is set)
```

**Key concept — staging area:** `git add` doesn't commit, it *stages*. `git commit` only includes what's staged. Skipping `git add` before `commit` means nothing gets committed.

**Key concept — git tracks files, not folders.** An empty folder is invisible to git. Workaround: add a placeholder file (convention: `.gitkeep`, empty, purely to give git something to track). Once a folder has real files in it, `.gitkeep` is no longer needed and can be deleted.

**`git status` habit:** Always check `git status` before `git add .` — it shows staged/unstaged/untracked files so you know exactly what you're about to commit, rather than blindly trusting the command.

## 2. .gitignore

**Why one root-level `.gitignore` instead of separate ones per subfolder?** Single place to maintain; avoids forgetting a rule in one of multiple files as the project grows.

**Root `.gitignore` content:**
```
# Dependencies
node_modules/

# Environment variables
.env
.env.local
.env*.local

# Next.js build output
frontend/.next/

# OS junk files
.DS_Store
Thumbs.db

# Editor
.vscode/
```

- `node_modules/` — huge, regenerable via `npm install`, never commit
- `.env` files — secrets (DB URLs, JWT secrets, API keys)
- `.next/` — Next.js build cache, regenerated on build
- OS junk files — auto-generated by the operating system, irrelevant to the project
- `.vscode/` — editor-specific settings, optional to ignore for solo projects

**Verification habit:** After `.gitignore` changes, run `git add .` and confirm `node_modules/` never appears in the file list — that's proof the ignore rules are working.

## 3. Monorepo Decision

**Structure chosen:**
```
EduSupply/
├── frontend/     (Next.js — deployed to Vercel)
├── backend/      (Express — deployed to Railway)
├── .gitignore    (root level)
└── README.md
```

**Reasoning:** Both Vercel and Railway support deploying from a subfolder within a single repo — no need for two separate repos. One repo is simpler to manage for a solo project of this scale.

**Backend choice — Express vs Next.js API routes:** Chose **Express**, reusing JWT auth + role-based access patterns already built in CampusOS, rather than rewriting that logic as Next.js API routes. Matches the stated deployment split (Vercel for frontend, Railway for backend/DB).

## 4. Frontend Setup (Next.js)

**Scaffold command:**
```bash
cd frontend
npx create-next-app@latest .
```
The trailing `.` scaffolds into the *current* folder instead of creating a new subfolder.

**Configuration chosen:**
| Option | Choice | Why |
|---|---|---|
| TypeScript | Yes | Type safety for data models (orders, products, inventory) — catches bugs before runtime, stronger portfolio signal |
| ESLint | Yes | Catches code errors/style issues early |
| Tailwind CSS | Yes | Per project spec |
| `src/` directory | Yes | Cleaner separation between app code and root config files |
| App Router | Yes | Per project spec |
| Import alias (`@/*`) | Yes | Cleaner imports |

**Verify it runs:**
```bash
npm run dev
```
Confirms the dev server starts (default `http://localhost:3000`) and the default Next.js page renders in-browser. Never trust a scaffold without visually verifying it.

**Note on `npm audit`:** Flagged 2 moderate vulnerabilities in a nested `postcss` dependency. The suggested `npm audit fix --force` would have downgraded Next.js to a 2019 version (breaking the entire App Router setup) — a classic case where blindly forcing an audit fix causes more damage than the vulnerability itself. Left alone for now; revisit later if it matters.

## 5. Backend Setup (Express)

**Scaffold commands:**
```bash
cd backend
npm init -y                  # generates package.json with defaults
npm install express pg jsonwebtoken bcrypt cors dotenv
npm install --save-dev nodemon
```

**Package purposes:**
| Package | Purpose |
|---|---|
| `express` | Web framework — routes, requests, responses |
| `pg` | PostgreSQL driver for Node |
| `jsonwebtoken` | Create/verify JWT auth tokens |
| `bcrypt` | Hash passwords before storing (never store plain text) |
| `cors` | Allows cross-origin requests (frontend on one port, backend on another) |
| `dotenv` | Loads `.env` variables into `process.env` |
| `nodemon` (dev dependency) | Auto-restarts server on file changes during development |

**Basic server (`backend/index.js`):**
```js
const express = require('express');

const app = express();

app.get('/', (req, res) => {
  res.send('Server is running');
});

app.listen(5000, () => {
  console.log('Server is running');
});
```

**Key syntax notes:**
- Plain Node (via `npm init`) defaults to **CommonJS** (`require(...)`), not ES Modules (`import ...`) — different from the Next.js/TypeScript frontend.
- `app.listen()` takes two arguments: a port number and a callback function — not a single string.
- `res.send()` sends the response body back to the client.

**`package.json` dev script added:**
```json
"scripts": {
  "dev": "nodemon index.js"
}
```
Run via `npm run dev`. Verified by checking both the terminal log *and* visiting `http://localhost:5000` in-browser — confirming the actual HTTP response, not just that the process started.

## Mistakes Made & Lessons Learned

1. **Accidentally deleted root `.gitignore` and both `.gitkeep` files** while troubleshooting a dependency install error — turned out `.gitkeep` wasn't actually the cause of that error (it's just an empty file, harmless to installs). Lesson: verify the actual cause of an error before deleting things reflexively.
2. **`dir` in PowerShell hides dotfiles by default** — use `dir -Force` to see hidden files like `.gitignore` when debugging what's actually on disk.
3. **Terminal working directory matters** — running `dir ..` from inside `eduSupply` vs from inside `frontend` gives completely different (and confusing) results if you lose track of where you are.
